# Security (plan): Candidate scope integrity

## Actor
Security (plan reviewer, claude-code harness) — adversarial security persona, distinct from the Architect.

## Threat model

**Subject of review.** The change removes the active change's eleven canonical process artifacts from the strict Candidate (the exact immutable subject Build/Security(code)/Test validate) and compensates with: a dedicated prose secret-scan lane (D4), a source-drift guard (D5), a mixed-landing pre-commit guard (D5b), a schema bump (D3), receipt-bound prose postimages + judgment-artifact digests (D6), and coherence diagnostics (D7). I verified every load-bearing code citation in `design.md` against the tree; all checked citations are accurate (candidate.rs:17, :365, :541-867, :905-922, :952-1027, :2433, :2534-2548, :2657-2675; closure.rs:561-599, :1726-1747, :2883-2895, :3128-3134, :3431-3438, :3577-3579; cli.rs:3304-3309, :3391-3495, :3559-3597, :5595-5611, :6882-6916; checks/mod.rs:85-219; ledger.rs:479-531, :970).

### Trust boundaries and assets
- **Asset 1 — secrets in prose.** A credential pasted into any of the 11 artifacts (by a persona, a paste accident, or a malicious brief) must be caught before it reaches a commit or a push.
- **Asset 2 — validated-code integrity.** The bytes the sandbox validated must be the bytes the landing commit ships (the d482a20 failure class: candidate.rs:958-980 silently routes undeclared dirty tracked files to `excluded` and pins base-HEAD bytes; closure.rs:3577-3579 silently skips the impure commit; closure.rs:3431-3438 returns `coherent:false` with empty blockers).
- **Asset 3 — archived prose integrity.** The artifact bytes that gated must be the bytes archived and committed.
- **Trust model.** Local `.mpd/` state, the ledger, and retained candidate sidecars are process-owned but not cryptographically protected from a local writer; the durable authorities are the candidate id (content-addressed), the archived closure plan, `scoped_commit_equivalence` at landing, and the commit/push scan floors. This change must not weaken any of those authorities.

### Prose secret-scan coverage: BEFORE vs AFTER (the central question)

| Lane | BEFORE | AFTER |
|---|---|---|
| Sandbox gitleaks (tool-locked, **required**) + semgrep over Candidate | Security(code) + Test profiles, over prose bytes **frozen at Build capture** (post-Build artifacts never enter the current drive's candidate) | Prose absent from the sandbox subject (D2/D9) |
| D4 prose lane (builtin + allowlist, fail-closed; gitleaks-dir additive) | — (does not exist) | Build execute, Build `--reuse`, Security(code) execute, Test execute, Test `--reuse` — **live worktree bytes**, path-addressed (untracked covered), symlinks fail closed via `secrets::scan_paths` (secrets.rs:399-405) |
| Staged pre-commit scan (`scan_staged_postimages`, checks/mod.rs:190-219 via cli.rs:5597-5611) | every staged prose path | **unchanged** (design pins with e2e) |
| Full `mpd check` tracked scan + ambient gitleaks/semgrep (cli.rs:5479-5543) | tracked prose | **unchanged** (D9) |
| Pre-push outgoing scan | outgoing prose | **unchanged** (D9) |

Net assessment: **coverage is wider in files, gates, and timing** — prose is now scanned at five strict gate sites (including both reuse paths, where today Build has *no* secret scan at all: the `build` profile runs format/clippy/tests/release only), scanned as live bytes (so post-Build judgment artifacts are scanned *before* commit for the first time), and untracked prose is covered. The union is complete: every path under `openspec/changes/<change>/` is either still Candidate-bound (`manifest.json`, `specs/**`, `history/**` — R6/D9, sandbox scans them) or one of the 11 (D4 lane scans them). At commit, `scan_staged_postimages` scans every staged path regardless of origin — the floor is intact.

Three genuine deltas remain, and they are the basis for my conditions:
1. **Semgrep-on-prose is lost — verified vacuous.** `security/semgrep/local-ci.yml` contains only `languages: [rust]` rules; it never matched `.md` content. Zero real loss.
2. **Gitleaks trust downgrade on prose.** The sandbox lane runs a tool-locked, digest-pinned, *required* gitleaks. The D4 lane runs an ambient-`PATH`, *optional* gitleaks (`run_tool` → `Command::new(name)`, checks/mod.rs:25-35; absence skipped-not-clean). On this repo, prose coverage by gitleaks silently degrades to builtin-only if gitleaks is missing or shadowed. → Condition 2.
3. **Gitleaks config-resolution evasion in the parameterized lane.** `run_gitleaks` keys its repo-owned-config decision off `root.join(".gitleaks.toml")` (checks/mod.rs:89) and scans `-s .`. Generalized to `-s openspec/changes/<change>`, gitleaks resolves `.gitleaks.toml`/`.gitleaksignore` **from the scanned source directory** when no `-c` is passed — so a planted `openspec/changes/<change>/.gitleaks.toml` (`useDefault = false`) silently blinds the additive gitleaks layer. The builtin floor is unaffected, but this must be closed. → Condition 1.

One narrow content-class delta: the builtin scanner applies content rules only to UTF-8 text; sandbox gitleaks would scan non-UTF-8 bytes. The 11 artifacts are canonical markdown with no legitimate binary case. → Condition 3.

### D4 fail-closed audit — sound as designed
Verified against real primitives: `scan_secrets` propagates every `scan_paths` error (checks/mod.rs:178-185); `scan_paths` refuses symlinks/non-regular/oversize/aggregate-overflow (secrets.rs:388-415); including symlinked prose in the enumerated set (rather than skipping it) therefore blocks the gate. Allowlist application mirrors the existing lane (suppressions always counted/printed; `.mpd/secret-allowlist.json` is version-controlled trust, and Security(code) — which always re-executes, cli.rs:3304-3309 — binds `AllowlistDigest`, closure.rs:1984). Call-site ordering is correct in the plan; the reuse path's `evaluate_strict_objective_reuse` performs the Build recapture internally (cli.rs:3168-3173), so the lane must sit before that call — the design places it exactly there.

### D5 drift guard — TOCTOU accepted; smuggling paths verified closed
The guard is advisory-refusal UX and races capture; acceptable because the integrity authorities are unchanged and independently sufficient: (a) the Candidate binds only declared paths, byte-frozen with intra-capture reobservation (candidate.rs:582-596); (b) D5b blocks a mixed landing commit at `git commit`; (c) `resolve_closure_landing`'s parent-diff purity + `scoped_commit_equivalence` refuse an impure landing at `publish --verify` regardless of hooks. **Untracked-warn abuse:** staging an untracked file converts it to an `A.` index entry (tracked bucket at any later gate); staged into the landing commit it fails D5b, and under `--no-verify` it fails landing purity at publish. Committing it *separately* stays possible — the pre-existing, doctrine-sanctioned user-owned lane, unchanged. **Exemption widening:** `mutable_process_path` is code-owned and Cond 7 forbids re-listing; a source file hidden under `.mpd/state/**` cannot join the landing commit because `allowed_paths` contains only the concrete ledger path (`closure::allowed`, closure.rs:3128-3134, no arm admits siblings of a concrete file) — D5b and coherence both refuse it.

### D5b — `--no-verify` bypass is acceptable
The hook is UX; `publish --verify` remains the authority (cli.rs:6384-6388 refuses on `!coherent`, and D7 guarantees non-empty blockers). Fail-closed requirements (unreadable ledger, missing `archive_closure`, malformed archive dir name → block) are in the design; rename entries must be checked on both sides (Condition 7).

### D3 schema bump — disjointness verified
`CANDIDATE_SCHEMA` is the first element of the id preimage tuple (candidate.rs:2664-2674), so v1/v2 ids are disjoint absent a SHA-256 collision; reopen refuses `version != CANDIDATE_SCHEMA` (candidate.rs:365). Build reuse recaptures fresh (v2 id ≠ v1 origin → refuse, cli.rs:3186-3192); Test reuse reopens the retained capture, which the version check refuses if v1. Stale v1 roots/records live at `<v1-id>` paths and are never compared. No cross-schema path found.

### D6 — self-attestation and the legacy-warn window
The digest is recorded by the `mpd` gate process from bytes just validated by `strict_artifact_issues` (cli.rs:4426-4442, symlink-refusing `read_contained`). Stripping the digest requires editing the ledger file — the same local-tamper class that could already forge receipts; the durable authority is the archive-time chain: gate-time bytes → (freshness keys for proposal/design/tasks/design-mock/documentation + D6 digests for the six judgment files) → archive-time worktree read (cli.rs:6910-6915) → plan postimage (duplicates refused by `claim_overlay`) → `scoped_commit_equivalence` at landing. This *restores* the byte-pin the Candidate previously provided for pre-Build prose and *extends* it to post-Build judgment artifacts that today have no protection at all. Acceptable per R5, with Condition 5. `verify_candidate_scope_entries` (closure.rs:1199-1239) checks candidate entries against the tree, not the reverse — prose absent from both sides passes without weakening the check.

### D7 / D8 — no security regression
D7 only adds diagnostics (paths, bounded, never contents) and removes the empty-blockers arm; `ready_to_commit` semantics preserved. D8's reuse firing set is safe: the reuse equality set is unchanged (candidate id covers all declared code/config via `entries_digest`; a committed edit moves `base_tree`); a prose edit that also sneaks a declared code change alters the id; an undeclared tracked change now refuses at the drift guard; an undeclared untracked file never enters the validated subject nor the landing commit. Security(code) categorically re-executes.

## Conditions for Builder

1. **Pin gitleaks config resolution in the parameterized prose lane.** `run_gitleaks(root, source_dir)` must ALWAYS pass an explicit `-c` — the repo-root `.gitleaks.toml` when present, else the ephemeral extend-default config (checks/mod.rs:75-76) — so gitleaks can never resolve configuration from the scanned change directory. Additionally treat the presence of `openspec/changes/<change>/.gitleaks.toml` or `.gitleaksignore` as a gate refusal in `enforce_prose_secret_scan`. Test: a rule-disabling `.gitleaks.toml` inside the change dir must NOT make a seeded prose secret scan clean.
2. **Gitleaks requiredness parity.** When strict `local_validation` config declares gitleaks `requirement: "required"` (as this repo does), the prose lane treats an unavailable gitleaks as a gate refusal, not a silent skip. Repos without that declaration keep the `run_external_scanners` skipped-not-clean semantic. Verify at code + test.
3. **Prose-lane stat and encoding fail-closed.** In `scan_change_prose`: only `NotFound` skips; every other `symlink_metadata` error is an `Err` (blocks). Non-UTF-8 bytes in any of the 11 canonical artifacts are a refusal, not a content-rules skip. Unit tests for both.
4. **Ordering proof for all five call sites.** The prose lane runs before any candidate capture/reopen/receipt evaluation at Build execute, Security(code) execute, Test execute, Build `--reuse`, and Test `--reuse` — before `evaluate_strict_objective_reuse`, whose Build arm recaptures internally (cli.rs:3168-3173). The secret-smuggling e2e must include the Test `--reuse` site, not only Build reuse.
5. **D6 digest integrity.** Compute the recorded SHA-256 from the SAME byte buffer `strict_artifact_issues` just validated (single read threaded to both consumers — a second read races a concurrent edit). At archive: a present-digest mismatch is always fatal; the absent-digest (legacy) warning prints phase and file; test that mismatch is never downgraded.
6. **Prune fail-closed on every non-file kind.** `prune_change_process_artifacts` errors (never skips) on a symlink AND on a directory/special file occupying one of the 11 paths in the base tree (an artifact-path directory is an error, not a recursive prune). Inventory/projection verification failing closed on a physically present artifact needs a direct unit test.
7. **D5b staged-path completeness.** Evaluate BOTH sides of rename/copy entries from `diff_cached_name_status` against `closure::allowed`; malformed archive dir name, unreadable `.mpd/state/<change>.json`, or missing `archive_closure` blocks the commit; only commits staging an `openspec/changes/archive/**` path are affected. e2e for the mixed-landing block and the malformed-name block.
8. **Schema-boundary tests.** Pin `CANDIDATE_SCHEMA == 2` in the id preimage and the reopen refusal (with remedy text); cross-schema reuse e2e: a v1 origin receipt under the new binary produces "candidate identity has changed" (fresh execution), never reuse, for both Build and Test.
9. **Exclusion-set single source of truth.** `change_process_artifact_paths` is consumed by `source_digest`, the candidate prune/overlay/inventory, AND `scan_change_prose`; the pinning test asserts the exact 11 filenames and byte-identical `source_digest` behavior before/after the refactor. Only the ACTIVE change's artifacts are excluded (sibling-change prose stays Candidate-bound) — test it.
10. **Commit-floor pinning e2e**: a secret staged in `design.md` (pre-Build artifact) and in `test.md` (post-Build artifact) is each caught by `scan_staged_postimages` via `mpd hook pre-commit`, independent of the gate lanes.

## Verdict

CONDITIONAL PASS

The design achieves its central security claim: with D4 in place, prose secret-scan coverage is strictly wider than today (five strict gate sites including both reuse paths vs two sandbox profiles; live bytes including post-Build judgment artifacts; untracked prose covered; commit and push floors unchanged), and no validation-evasion path is opened — the Candidate still binds all code/config, Security(code) always re-executes, D5b plus landing purity/equivalence close the d482a20 class, and the v1/v2 schema boundary is fail-closed. The conditions above close the only real deltas found: gitleaks config-resolution evasion (Condition 1), the required→optional gitleaks downgrade on prose (Condition 2), and fail-closed edges in the lane, prune, digest, and hook (Conditions 3-10). Owner: Builder, verified at Security (code) with the tests named per condition. Unresolved conditions block deployment; this is High-risk novel surface, so findings at Security (code) are not fixed inline — re-run the gate after every fix.
